# <a name="project-migration-and-upgrade-reference-for-visual-studio"></a>Odwołanie do migracji i uaktualnienia projektu dla programu Visual Studio

::: moniker range="vs-2017"

Każda wersja programu Visual Studio zazwyczaj obsługuje większość poprzednich typów projektów, plików i innych zasobów. Można pracować z [nimi, jak zawsze](../ide/solutions-and-projects-in-visual-studio.md), i pod warunkiem, że nie zależy od nowszych funkcji, Visual Studio próbuje zachować zgodność wstecz z poprzednich wersji, takich jak Visual Studio 2015, Visual Studio 2013 i Visual Studio 2012. (Zobacz [informacje o wersji,](/visualstudio/releasenotes/vs2017-relnotes/) dla których funkcje są specyficzne dla których wersji.)

Obsługa niektórych typów projektów również zmienia się wraz z czasem. Nowsza wersja programu Visual Studio może nie obsługiwać niektórych projektów w ogóle lub wymaga aktualizacji projektu w taki sposób, że nie jest już wstecznie zgodne. Aby uzyskać aktualny stan problemów z migracją, zobacz [witrynę społeczności deweloperów programu Visual Studio](https://developercommunity.visualstudio.com).

Ten bieżący artykuł zawiera szczegółowe informacje tylko dla typów projektów, które visual studio 2017 może migrować. Artykuł nie obejmuje typów projektów, które nie są już obsługiwane w programie Visual Studio 2017 i dlatego nie można ich przeprowadzić do migracji. Artykuł wyklucza również obsługiwane typy projektów, które nie mają problemów z migracją; lista ta znajduje się na [platformie Targeting and Compatibility](/visualstudio/productinfo/vs2017-compatibility-vs).

> [!IMPORTANT]
> Niektóre typy projektów wymagają zainstalowania odpowiednich obciążeń za pośrednictwem instalatora programu Visual Studio. Jeśli nie masz zainstalowanego obciążenia, visual studio zgłasza nieznany lub niezgodny typ projektu. W takim przypadku sprawdź opcje instalacji i spróbuj ponownie. Ponownie zobacz artykuł [o kierowaniu i zgodności platformy,](/visualstudio/productinfo/vs2017-compatibility-vs) aby uzyskać szczegółowe informacje na temat obsługi projektu w programie Visual Studio 2017.

## <a name="how-visual-studio-decides-when-to-migrate-a-project"></a>Jak program Visual Studio decyduje o tym, kiedy przeprowadzić migrację projektu

Każda nowa wersja programu Visual Studio zazwyczaj ma na celu zachowanie zgodności z poprzednimi wersjami, tak aby ten sam projekt można otworzyć, zmodyfikować i skompilować w różnych wersjach. Jednak z czasem zachodzą nieuniknione zmiany, w związku z którymi niektóre typy projektów mogą nie być już obsługiwane. (Zobacz [kierowanie na platformę i zgodność,](/visualstudio/productinfo/vs2017-compatibility-vs) dla których typy projektów są obsługiwane w programie Visual Studio 2017. W takich przypadkach nowsza wersja programu Visual Studio nie ładuje projektu i nie oferuje ścieżki migracji; należy zachować tego projektu w poprzedniej wersji programu Visual Studio, który obsługuje go.

W innych przypadkach nowsza wersja programu Visual Studio może otworzyć projekt, ale musi zaktualizować lub przeprowadzić migrację projektu w taki sposób, aby mógł uczynić go niezgodnym z poprzednimi wersjami. Visual Studio używa szeregu kryteriów, aby ustalić, czy taka migracja jest konieczna:

- Zgodność z docelowych wersji platform, powrót do programu Visual Studio 2013 RTM.

- Zgodność zasobów w czasie projektowania z poprzednimi wersjami programu Visual Studio. (Mianowicie różne kanały programu Visual Studio 2017; Visual Studio 2015 RTM & aktualizacji 3; Visual Studio 2013 RTM & aktualizacji 5; Aktualizacja programu Visual Studio 2012 4; Visual Studio 2010 SP 1.) Visual Studio 2017 ma na celu niepowodzenie bezpiecznie z przestarzałych zasobów czasu projektowania bez ich uszkodzenia, tak, że poprzednie wersje nadal można otworzyć projekt.

- Czy nowe zasoby czasu projektowania spowoduje przerwanie zgodności z poprzednimi wersjami do programu Visual Studio 2013 RTM & aktualizacji 5.

Właściciel inżynierii danego typu projektu analizuje te kryteria i wykonuje wezwanie w przypadku pomocy technicznej, zgodności i migracji. Ponownie visual studio próbuje zachować zgodność przezroczyste między wersjami programu Visual Studio, jeśli to możliwe, co oznacza, że można tworzyć i modyfikować projekty w jednej wersji programu Visual Studio i po prostu działa w innych wersjach.

Jeśli taka zgodność nie jest jednak możliwe, podobnie jak w przypadku niektórych typów projektu opisanych w tym artykule, program Visual Studio otwiera kreatora uaktualniania, aby wprowadzić niezbędne zmiany jednokierunkowe.

Takie jednokierunkowe zmiany mogą `ToolsVersion` obejmować zmianę właściwości w pliku projektu, co oznacza dokładnie, która wersja MSBuild można włączyć kodu źródłowego projektu do nierozerwalnie i wdrażalne artefakty, które ostatecznie chcesz. Oznacza to, że co renderuje projekt niezgodny z poprzednimi wersjami programu Visual Studio nie jest `ToolsVersion`wersja programu Visual *Studio,* ale wersja *MSBuild,* zgodnie z ustaleniami . Tak długo, jak wersja programu Visual Studio zawiera msbuild toolchain, który pasuje `ToolsVersion` do w projekcie, a następnie Visual Studio można wywołać tego pasowania narzędzi do tworzenia projektu.

Aby zachować maksymalną zgodność z projektami utworzonymi w starszych wersjach, visual studio 2017 zawiera niezbędne narzędzia MSBuild do obsługi `ToolsVersion` 15, 14, 12 i 4. Projekty, które używają `ToolsVersion` dowolnej z tych wartości powinny spowodować pomyślną kompilację. (Ponownie temat, czy visual studio 2017 obsługuje typ projektu w ogóle, zgodnie z opisem w określaniu kierowania na [platformę i zgodności](/visualstudio/productinfo/vs2017-compatibility-vs).)

W tym kontekście naturalnie pojawia się pytanie, czy należy spróbować ręcznie zaktualizować `ToolsVersion` lub przeprowadzić migrację projektu do nowszej wartości. Wprowadzenie takiej zmiany jest niepotrzebne i prawdopodobnie wygenerowałoby wiele błędów i ostrzeżeń, które należy naprawić, aby projekt został ponownie skompilowy. Ponadto jeśli visual studio porzuci `ToolsVersion` obsługę dla określonego w przyszłości, a następnie otwarcie projektu `ToolsVersion` spowoduje proces migracji projektu specjalnie dlatego, że wartość musi zostać zmieniona. W takim przypadku podsystem dla tego określonego typu projektu wie dokładnie, co należy zmienić i może automatycznie wprowadzać te zmiany, jak opisano wcześniej w tym artykule.

## <a name="migrate-a-project"></a>Migrowanie projektu

Chociaż staramy się zachować zgodność z poprzednimi wersjami, mogą wystąpić zmiany, które nie są zgodne z poprzednimi wersjami. (Zobacz [kierowanie na platformę i zgodność,](/visualstudio/releases/2019/compatibility) dla których typy projektów są obsługiwane w programie Visual Studio 2019. W takim przypadku nowsza wersja programu Visual Studio nie załaduje projektu ani nie zaoferuje ścieżki migracji. Może być trzeba zachować tego projektu w poprzedniej wersji programu Visual Studio.

Czasami nowsza wersja programu Visual Studio może otworzyć projekt, ale musi zaktualizować lub przeprowadzić migrację projektu w sposób, który może spowodować jego niezgodność z poprzednimi wersjami. Visual Studio używa szeregu kryteriów, aby ustalić, czy taka migracja jest konieczna:

- Zgodność z docelowych wersji platform, powrót do programu Visual Studio 2013 RTM.

- Zgodność zasobów w czasie projektowania z poprzednimi wersjami programu Visual Studio. (Mianowicie różne kanały programu Visual Studio 2019, Visual Studio 2017; Visual Studio 2015 RTM & aktualizacji 3; Visual Studio 2013 RTM & aktualizacji 5; Aktualizacja programu Visual Studio 2012 4; Visual Studio 2010 SP 1.) Visual Studio 2019 ma na celu niepowodzenie bezpiecznie z przestarzałych zasobów czasu projektowania bez ich uszkodzenia, tak, że poprzednie wersje nadal można otworzyć projekt.

- Czy nowe zasoby czasu projektowania spowoduje przerwanie zgodności z poprzednimi wersjami do programu Visual Studio 2013 RTM & aktualizacji 5.

Zespół inżynierów, który jest właścicielem typu projektu, analizuje te kryteria i wykonuje wywołanie, jeśli chodzi o obsługę, zgodność i migrację. Ponownie staramy się zachować zgodność między wersjami programu Visual Studio, tak aby podczas tworzenia i modyfikowania projektów w jednej wersji programu Visual Studio, po prostu działa w innych wersjach.

Czasami zgodność nie jest możliwa. Następnie program Visual Studio otwiera kreatora uaktualniania, aby wprowadzić niezbędne zmiany jednokierunkowe. Te jednokierunkowe zmiany mogą `ToolsVersion` obejmować zmianę właściwości w pliku projektu, który oznacza dokładnie, która wersja MSBuild można włączyć kod źródłowy projektu do artefaktów, które można uruchomić i wdrożyć, które chcesz. 

Co renderuje projekt niezgodny z poprzednimi wersjami programu Visual Studio nie jest wersja programu `ToolsVersion`Visual *Studio,* ale wersja *MSBuild,* zgodnie z ustaleniami . Jeśli wersja programu Visual Studio zawiera pasek narzędzi MSBuild, który pasuje do `ToolsVersion` w projekcie, a następnie Visual Studio można wywołać tego pasowania narzędzi do tworzenia projektu.

Aby zachować zgodność z projektami utworzonymi w poprzednich wersjach, visual studio 2019 zawiera `ToolsVersion` niezbędne narzędzia MSBuild do obsługi 15, 14, 12 i 4. Projekty, które używają `ToolsVersion` dowolnej z tych wartości powinny spowodować pomyślną kompilację. (Ponownie temat, czy visual studio 2019 obsługuje typ projektu, zgodnie z opisem w [określaniu kierowania na platformę i zgodności](/visualstudio/releases/2019/compatibility).)

Możesz być kuszony, aby ręcznie zaktualizować lub przenieść projekt do nowszej `ToolsVersion` wartości. Nie jest konieczne, aby dokonać takiej zmiany i prawdopodobnie wygenerować wiele błędów i ostrzeżeń, które należy naprawić, aby uzyskać projekt do utworzenia ponownie. Ponadto jeśli program Visual Studio nie `ToolsVersion` obsługuje określonego w przyszłości, a następnie projekt wyzwala proces `ToolsVersion` migracji projektu po otwarciu go, ponieważ jego wartość musi zostać zmieniona.
