# <a name="c26447-dontthrowinnoexcept"></a>C26447 DONT_THROW_IN_NOEXCEPT

Funkcja jest zadeklarowana **noexcept** , ale wywołuje funkcję, która może zgłaszać wyjątki.

C++ podstawowych wytycznych dotyczących języka: [F.6: Jeśli funkcja nie może zgłaszać, Zadeklaruj go noexcept](https://github.com/isocpp/CppCoreGuidelines/blob/master/CppCoreGuidelines.md#f6-if-your-function-may-not-throw-declare-it-noexcept).

Ta reguła zmienia inną regułę [C26440 DECLARE_NOEXCEPT](c26440.md), który podejmie próbę odnalezienia funkcje, które dobrze nadają się być oznaczony jako **noexcept**. W tym przypadku chodzi o to, że gdy niektórych funkcji jest oznaczony jako **noexcept**, jego umowy muszą być przechowywane przez nie wywoływania innego kodu, który może zgłaszać wyjątki.

- Kompilator Microsoft C++ już obsługuje proste naruszenia, takich jak **throw** instrukcji w treści funkcji (zobacz [C4297](/cpp/error-messages/compiler-warnings/compiler-warning-level-1-c4297)).
- Reguła dotyczy tylko wywołania funkcji. Jego flagi elementów docelowych, które nie są **constexpr** i który potencjalnie może generować wyjątki; innymi słowy nie oznaczonych jawnie jako niezgłaszające przy użyciu **noexcept**, **__declspec () nothrow)**, **throw()**.
- Funkcje generowane przez kompilator docelowej są pomijane redukcji szumu, ponieważ specyfikacje wyjątków nie są zawsze dostarczane przez kompilator.
- Moduł sprawdzania pomija również specjalnymi rodzajami funkcji docelowego, które powinny zostać wdrożone jako **noexcept**; ta zasada jest wymuszana przez [C26439 SPECIAL_NOEXCEPT](c26439.md).
