# <a name="ca1031-do-not-catch-general-exception-types"></a>CA1031: Nie przechwytuj wyjątków typów ogólnych

|||
|-|-|
|TypeName|DoNotCatchGeneralExceptionTypes|
|CheckId|CA1031|
|Kategoria|Microsoft.Design|
|Zmiana podziału|Nieprzerwanie|

## <a name="cause"></a>Przyczyna
Ogólny wyjątek, taki jak <xref:System.Exception?displayProperty=fullName> lub <xref:System.SystemException?displayProperty=fullName>, jest przechwytywany w instrukcji `catch` lub ogólnej klauzuli catch, takiej jak `catch()` jest używany.

Domyślnie ta reguła flaguje tylko ogólne typy wyjątków, które są przechwytywane, ale [można je skonfigurować](#configurability).

## <a name="rule-description"></a>Opis reguły
Ogólne wyjątki nie powinny być przechwytywane.

## <a name="how-to-fix-violations"></a>Jak naprawić naruszenia
Aby naprawić naruszenie tej zasady, Przechwyć bardziej szczegółowy wyjątek lub ponownie Zgłoś wyjątek ogólny jako ostatnią instrukcję w bloku `catch`.

## <a name="when-to-suppress-warnings"></a>Kiedy pominąć ostrzeżenia
Nie pomijaj ostrzeżeń dla tej reguły. Przechwytywanie typów wyjątków ogólnych może spowodować ukrycie problemów w czasie wykonywania z poziomu użytkownika biblioteki i utrudnienie debugowania.

> [!NOTE]
> Począwszy od .NET Framework 4, środowisko uruchomieniowe języka wspólnego (CLR) nie dostarcza już wyjątków uszkodzonych stanu występujących w systemie operacyjnym i kodzie zarządzanym, takich jak naruszenia dostępu w [!INCLUDE[TLA#tla_mswin](../code-quality/includes/tlasharptla_mswin_md.md)], do obsługi przez kod zarządzany. Jeśli chcesz skompilować aplikację w .NET Framework 4 lub nowszej wersji i zachować obsługę wyjątków uszkodzonych Stanów, można zastosować atrybut <xref:System.Runtime.ExceptionServices.HandleProcessCorruptedStateExceptionsAttribute> do metody, która obsługuje wyjątek uszkodzonego stanu.

## <a name="configurability"></a>Określając

Jeśli uruchamiasz tę regułę z [analizatorów FxCop](install-fxcop-analyzers.md) (a nie ze starszą analizą), tę regułę można skonfigurować.

### <a name="disallowed-exception-type-names"></a>Niedozwolone nazwy typów wyjątków

Można skonfigurować, które typy wyjątków są niedozwolone. Na przykład, aby określić, że reguła powinna flagować `catch` obsługi przy użyciu `NullReferenceException`, Dodaj następującą parę klucz-wartość do pliku editorconfig w projekcie:

```ini
dotnet_code_quality.CA1031.disallowed_symbol_names = NullReferenceException
```

Dozwolone formaty nazw typów w wartości opcji (oddzielone znakiem "|"):
  - Tylko nazwa typu (zawiera wszystkie symbole z nazwą, niezależnie od typu zawierającego lub przestrzeni nazw)
  - W pełni kwalifikowane nazwy w [formacie identyfikatora dokumentacji](https://github.com/dotnet/csharplang/blob/master/spec/documentation-comments.md#id-string-format) symbolu z prefiksem `T:`.

Przykłady:

| Wartość opcji | Podsumowanie |
| --- | --- |
|`dotnet_code_quality.CA1031.disallowed_symbol_names = ExceptionType` | Dopasowuje wszystkie symbole o nazwie "ExceptionType" w kompilacji
|`dotnet_code_quality.CA1031.disallowed_symbol_names = ExceptionType1|ExceptionType2` | Dopasowuje wszystkie symbole o nazwie "ExceptionType1" lub "ExceptionType2" w kompilacji
|`dotnet_code_quality.CA1031.disallowed_symbol_names = T:NS.ExceptionType` | Dopasowuje określone typy o nazwie "ExceptionType" przy użyciu podanej w pełni kwalifikowanej nazwy.
|`dotnet_code_quality.CA1031.disallowed_symbol_names = T:NS1.ExceptionType1|T:NS1.ExceptionType2` | Dopasowuje typy o nazwach "ExceptionType1" i "ExceptionType2" z odpowiednimi w pełni kwalifikowanymi nazwami

Można skonfigurować te opcje tylko dla tej reguły, dla wszystkich reguł lub dla wszystkich reguł w tej kategorii (projekt). Aby uzyskać więcej informacji, zobacz [Konfigurowanie analizatorów FxCop](configure-fxcop-analyzers.md).
